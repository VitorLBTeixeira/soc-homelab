# Splunk Setup — SOC Home Lab

Setup notes for the Splunk Enterprise indexer and Universal Forwarder configuration used in this lab.

---

## Splunk Enterprise (Indexer)

**Host:** Ubuntu 24.04 Server (VirtualBox VM)  
**Version:** Splunk Enterprise 10.x  
**Listening port:** 9997 (receiving)

### Installation (non-root — Splunk best practice)

> **Do not run Splunk as root.** The `.deb` package already creates the `splunk`
> user/group and installs to `/opt/splunk` owned by it. The mistake to avoid is
> running the **first `start` as root**: that creates files (`var/`, `splunk.secret`,
> indexes) owned by root inside `/opt/splunk`, produces mixed ownership, and splunkd
> will then only start with `--run-as-root` — a crutch, not a fix.

```bash
# 0. Download Splunk Enterprise (.deb) from the official site (requires a free
#    Splunk account): https://www.splunk.com/en_us/download/splunk-enterprise.html

# 1. Install the local package. 'dpkg -i' only INSTALLS a local file (it does not
#    download). 'apt install ./file.deb' is preferred — it resolves dependencies.
sudo apt install ./splunk-*.deb          # or: sudo dpkg -i splunk-*.deb

# 2. Make sure ALL of /opt/splunk is owned by the splunk user BEFORE the first start:
sudo chown -R splunk:splunk /opt/splunk

# 3. Let Splunk generate the systemd unit, running as the splunk user:
sudo /opt/splunk/bin/splunk enable boot-start -user splunk -group splunk -systemd-managed 1 --accept-license

# 4. Start via systemd (the generated unit is named Splunkd.service):
sudo systemctl daemon-reload
sudo systemctl start Splunkd

# 5. Confirm the process runs as splunk, NOT as root:
ps -u splunk -o pid,cmd | grep splunkd
systemctl status Splunkd
```

The unit generated by `enable boot-start -systemd-managed 1` ships hardened with
`NoNewPrivileges=yes` and `AmbientCapabilities=CAP_DAC_READ_SEARCH`. That capability
lets the **unprivileged** `splunk` process read protected logs (e.g. `/var/log`)
without root — which is exactly why people reach for `--run-as-root`. So **do not
hand-write the systemd unit**: letting Splunk generate it is both more correct and
more secure.

> **Recovery** (if the instance already started as root): `splunk stop` → kill any
> remaining `splunkd` running as root → `sudo chown -R splunk:splunk /opt/splunk` →
> `splunk disable boot-start` → repeat step 3 → check `SPLUNK_OS_USER=splunk` in
> `/opt/splunk/etc/splunk-launch.conf`.

### Enable receiving

In Splunk Web → Settings → Forwarding and Receiving → Configure Receiving → Add port **9997**.

## Universal Forwarder (Windows)

**Installed on:** Windows 10 Pro, Windows Server 2022 (DC-01)  
**Version:** Splunk Universal Forwarder 9.x

### Configuration

After installation, verify `outputs.conf` points to the correct indexer IP:

```
C:\Program Files\SplunkUniversalForwarder\etc\system\local\outputs.conf
```

```ini
[tcpout:default-autolb-group]
server = <SPLUNK_SERVER_IP>:9997
```

Restart the forwarder service after any changes:

```powershell
Stop-Service SplunkForwarder
Start-Service SplunkForwarder
```

## Windows Audit Policy (GPO)

Security event logging is **disabled by default** on Windows. Audit policy must be explicitly enabled.

### Windows 10 (local — gpedit.msc)

`Local Computer Policy → Computer Configuration → Windows Settings → Security Settings → Local Policies → Audit Policy`

Enable **Success + Failure** for:
- Audit logon events
- Audit account logon events

### Domain Controller (gpmc.msc)

`Forest → Domains → soc.lab → Domain Controllers → Default Domain Controllers Policy → Edit`

`Computer Configuration → Policies → Windows Settings → Security Settings → Local Policies → Audit Policy`

Enable **Success + Failure** for:
- Audit account logon events
- Audit logon events
- Audit account management
- Audit directory service access
- Audit object access

Apply immediately:
```cmd
gpupdate /force
```
